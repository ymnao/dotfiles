  2 体を並列で競わせ、指摘をマージして報告する。通常のレビューは code-reviewer
  単体か /codex-review を使い、これは「見逃しが許されない変更」専用。
---

現在の変更に対して、独立した 2 体のレビュアーを競争させて指摘の網羅性を上げる。
根拠: 競争フレーミングは報酬の実体がなくても指摘の深さ・数を増やす
(Jesse Vincent, 2026-05)。異種併用レビューでは指摘の 93.4% が単一レビュアー
のみ検出 — レビュアーを増やすこと自体に価値がある。

## 手順

1. レビュー対象を確定する: `git status --porcelain` と直近の diff を確認し、
   レビュー範囲(コミット済み diff か、作業ツリーの変更か)をユーザーの依頼から
   特定する。不明なら質問せず「ベース〜HEAD + 未コミット変更の全部」とする
2. **code-reviewer サブエージェントを 2 体、並列で起動する。**
   両方に同じレビュー範囲を与え、以下のプロンプトをそれぞれの依頼文の
   末尾に付ける(この文言をそのまま使う):
   - レビュアー A(正しさレンズ):
     「あなたはレビュアー A。別のレビュアー B と競争している。深刻な問題
     (Critical/Warning)をより多く・より正確に見つけた方に 5 ポイントが
     入る。特にロジックの誤り・境界条件・エラーハンドリング・互換性違反を
     深く掘ること。誤指摘はマイナス 1 ポイント。」
   - レビュアー B(セキュリティ・堅牢性レンズ):
     「あなたはレビュアー B。別のレビュアー A と競争している。深刻な問題
     (Critical/Warning)をより多く・より正確に見つけた方に 5 ポイントが
     入る。特に秘密情報の露出・入力検証・インジェクション・テスト欠如・
     既存抽象の迂回を深く掘ること。誤指摘はマイナス 1 ポイント。」
3. 両者の結果を統合する:
   - 同一指摘(同じファイル・同じ問題)は 1 件にまとめ、`[両者検出]` を付ける
   - 片方のみの指摘は `[A のみ]` / `[B のみ]` を付ける
   - 明らかな誤指摘(コードを読めば反証できるもの)は自分で該当箇所を Read
     して検証し、誤りなら除外して除外理由を記録する
4. 報告する。形式は code-reviewer の出力契約(Critical/Warning/Suggestion +
   file:line + 根拠)に合わせ、末尾に以下を付ける:
   - 検出数サマリ(A / B / 両者 / 除外した誤指摘数)
   - 「勝者」の宣言(深刻な指摘の正味数で判定)— 遊びではなくどのレンズが
     効いたかの記録として書く

## 使いどころ(重要)

- 対象: リスク分類 high の変更、セキュリティ境界(hooks / 認証 / 権限)を
  触る変更、リリース前の最終確認
- **毎 PR には使わない**。通常は code-reviewer 単体または /codex-review。
  これはコストが約 2 倍かかる代わりに見逃しを減らす道具
- codex-review(異種モデル)とは併用できる: 異種の意見が欲しければ
  /codex-review、同一モデル内で網羅性を上げたければこれ

## 報告前の照合

各指摘がサブエージェントの実際の出力に対応しているか照合すること。
統合時に自分で創作した指摘を混ぜない。実行していないレビューを実行したと
書かない。
