学術原稿を観点別サブエージェントの fan-out で検出 → メインが一次ソース・実装に
当たって検証 → ユーザー承認 → 適用 → ビルド確認、の順で添削する。実際の校正
セッションで引用文献の fabrication(実在する著者・媒体・年に偽の題名)を 2 件
検出した実績のある観点構成に基づく。検出した指摘をそのまま信じないこと —
レビュー側も fabrication し得るため、修正前の検証フェーズが本 skill の中核。

## 引数

- **必須**: 対象原稿ファイルのパス(複数可。`.tex` / `.md`)
- **任意**: 観点名(下表の Name)を並べると絞り込み実行(例: `/paper-review paper.tex facts claims`)
- **任意**: `--fix` — 検証済み指摘を承認を待たず自動適用する(既定はユーザー承認制)

## Pre-conditions

1. 対象ファイルが存在すること。なければ報告して停止
2. git リポジトリ内であることを推奨(カテゴリ別コミットのため)。リポジトリ外なら
   コミットせず working tree への適用のみ行う旨をユーザーに伝えて続行
3. **モデル分担**(生成者とレビュアーの分離。詳細はユーザー CLAUDE.md
   「モデル分担ルール」と dotfiles の docs/ai-operations.md §1): この skill は
   レビュアー側。原稿を主に書いたモデルと同一系統のセッションで実行すると盲点が
   重なる。原稿が主に Claude 生成なら別系統(codex 等)での追加レビューを最後に
   提案すること

## 観点(perspective)

| Name | Focus | 詳細チェックリスト |
|------|-------|--------------------|
| `facts` | 引用文献の実在・帰属の検証、本文の数値とデータ・実装の突合 | references/facts.md |
| `claims` | 主張と証拠のギャップ、過大表現、自己矛盾、hedging 強度 | references/claims.md |
| `methods` | 検定と実験設計の整合、擬似反復、効果量・CI、仮説の検証可能性 | references/methods.md |
| `terms` | 独自造語、未定義語・記号、表記統一、英文の冠詞・時制 | references/terms.md |
| `structure` | 概念の初出順、論理の流れ、abstract、研究計画の頑健性 | references/structure.md |
| `format` | 学会フォーマット、図表、数式書体、ページ制約、和文組版 | references/format.md |

既定は全 6 観点。ユーザーが観点名を指定したらそれのみ実行する。

観点間で同一の問題を見つけた場合は、上表の Focus が最も近い観点が報告する
(迷ったら重複報告してよい — 手順 3 の検証で濾し、提示時に 1 件にまとめる)。

## 手順

### 1. 準備

1. 原稿全体を Read する
2. リポジトリ内の突合対象を特定する: 結果ファイル(CSV 等)、設定ファイル
   (config.toml 等)、実装コード、`.bib` / thebibliography、実験ノート
3. ビルドコマンドを検出する: `.vscode/settings.json`(latex-workshop 設定)→
   `Makefile` → `latexmkrc` の順に探し、なければ `latexmk -pdf`(LaTeX)/
   ビルドなし(Markdown)を既定とする
4. 原稿の言語(和文/英文)と形式(LaTeX/Markdown)を判定し、観点への指示に含める
5. 原稿の性質から明らかに空振りになる観点(引用文献ゼロなら `facts` の引用検証、
   統計的検定なしなら `methods` の大半)があれば、除外を提案してユーザーに確認
   する(`--fix` 時は自動除外して報告に明記)

### 2. 検出(観点ごとにサブエージェント fan-out)

対象観点ごとに **Sonnet 世代のサブエージェントを並列で起動**する(fan-out は
中モデルに委譲するモデル分担ルール)。各エージェントへの依頼文に含めるもの:

- 原稿ファイルの絶対パス。手順 1 で特定した突合対象ファイルのパスは
  `facts` / `methods` にのみ渡す(他の観点は原稿だけで完結する。`format` には
  見本 PDF・投稿規定があれば追加で渡す)— 無関係なファイルを全観点に渡すと
  トークンを浪費する
- `$HOME/.claude/skills/paper-review/references/<観点>.md` を最初に読む指示
- 原稿の言語・形式
- `facts` には WebSearch / WebFetch で一次ソース(原論文の abstract、公式 PDF)に
  当たる指示を明示する。`terms` は独自造語の使用実績確認に限り WebSearch 可。
  他の観点はリポジトリ内ファイルのみで完結させる
- 出力契約: 下の「指摘フォーマット」で全件返す。見つからなければ「指摘なし」。
  over-report してよい(下流の検証フェーズで濾す)と伝える

### 3. 検証(全指摘、修正前に必ず)

**外部レビューも自分の fan-out も鵜呑みにしない。** 各指摘についてメインが
反証を試みる:

- `facts` の指摘 → 一次ソース(URL・PDF)またはリポジトリ内の実データ・実装を
  自分で開いて確認する。サブエージェントの「確認した」という報告だけで
  CONFIRMED にしない
- その他の観点 → 指摘が文脈の誤読・既対応・スタイルの好みでないか判定する。
  原稿は手順 1 で全文 Read 済みなので読み込み済みの内容で判定し、再 Read は
  行番号の確認などピンポイントに留める
- 判定: **CONFIRMED** / **REFUTED**(一行の理由を記録)。迷ったら CONFIRMED の
  まま重大度を下げてユーザーに委ねる

### 4. 提示と承認

CONFIRMED の全指摘を重大度順(critical → major → minor)に指摘フォーマットで
提示する。critical(fabrication・自己矛盾・数値と実データの不一致・検定と設計の
不一致)は先頭に置き、根拠の一次ソースを必ず添える。

- 既定: ユーザーの承認を待つ。「どれを適用するか」の部分選択も受け付ける
- `--fix` 指定時: 承認を待たず全 CONFIRMED を適用する。ただし critical は
  修正内容が原文の意味を変えるため、適用後の報告で個別に目立たせる

### 5. 適用

- 修正は**観点(カテゴリ)別にコミットを分け**、コミットメッセージ body に
  「指摘 → 修正」の対応を 1 件ずつ書く
- 紙面制約・スコープ外で対応できない指摘は捨てず、「スライドで / 本編で /
  別 issue で」と**対応先を明記**して報告表に残す
- ページ制約下の圧縮では意味・数値・限定表現を保持したまま助詞・冗長句を削る
  (段抜き表のセル圧縮は両カラム分効く)。限定表現(「探索した範囲では」等)を
  圧縮の犠牲にしない

### 6. ビルド確認

LaTeX なら手順 1 で検出したコマンドでリビルドし、エラーゼロと**ページ数が制約内
であること**を確認してから完了と言う。追記した場合は特にページ数を再確認する。
ビルドできない環境ならその旨を報告に明記する(できたと書かない)。

## 指摘フォーマット

```
#<N> [<critical|major|minor>] [<観点>] <file>:<line>
現行: <現行の文・語句>
問題: <何が・なぜ問題か>
修正案: <before> → <after>
根拠: <一次ソース URL / リポジトリ内 file:line / ガイドライン名>
```

重大度の基準:
- **critical**: fabrication(引用の実在・帰属の誤り)、本文の数値と実データの
  不一致、検定と実験設計の不一致、自己矛盾
- **major**: 過大主張、擬似反復、未定義の用語・記号、論理の断絶
- **minor**: 表記揺れ、組版、冗長表現

## 報告フォーマット

まずサマリ表:

| 観点 | 検出 | CONFIRMED | REFUTED | 適用 | 対応しない | 対応先送り |
|------|------|-----------|---------|------|-----------|------------|

次に全指摘への回答表(REFUTED 含む全件。1 件も落とさない):

| # | 指摘(file:line — 要約) | 判定 | 対応 |
|---|------------------------|------|------|
| 1 | ... | CONFIRMED | 修正済み(<コミットハッシュ>) |
| 2 | ... | CONFIRMED | 対応しない — <理由> |
| 3 | ... | CONFIRMED | 先送り — <スライドで/本編で/issue #N> |
| 4 | ... | REFUTED | <一行の反証理由> |

最後にビルド結果(コマンド・エラー有無・ページ数)。実行していない検証を
実行したと書かない。

## Notes

- **検証フェーズの実例**: 外部レビューの指摘を一次ソースで検証した結果、指摘側が
  すべて正しかった回もあれば、原稿側の引用が fabrication だった回もある
  (実在する著者・媒体・年に偽の題名、原論文の abstract と正反対の帰属)。
  どちらも通読では気づけず、一次ソースとの突合でのみ判定できた
- **コスト**: 全 6 観点の fan-out + facts の Web 検証で時間がかかる。長い原稿
  (卒論本体等)では章単位の分割実行か観点絞り込みをユーザーに提案する
- **コミットはするが push / PR はしない**(原稿リポジトリの運用はユーザーに従う)
