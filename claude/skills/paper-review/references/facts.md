# facts — ファクト・引用検証

最重要観点。もっともらしく見える記述ほど疑う。全指摘に一次ソース(URL または
リポジトリ内 file:line)を添えること。根拠を提示できない指摘は出さない。

## 1. 引用文献の実在・書誌情報

参考文献リストの各エントリについて WebSearch / WebFetch で一次ソースに当たる:

- **題名は実在するか**: 実在する著者・媒体・年に「もっともらしい偽の題名」が
  合成される fabrication が実際に起きる(実例: JPX ワーキングペーパーの実物と
  題名が異なっていた)。著者名・年だけの一致で安心しない
- 著者名の綴り、誌名・巻号・年、DOI の整合
- 孫引きの検出: 本文の記述が原典でなくレビュー論文・教科書経由になっていないか

検証の優先順位(長い原稿で全件の一次ソース精読は破綻する):

1. 同一文献の重複を除いたうえで、定量的主張・強い帰属(「〜を実証した」等)の
   直接根拠になっている引用を優先し、原論文の abstract まで確認する
2. 背景引用(「近年議論されている」等)は書誌情報(著者・題名・媒体・年)の
   実在確認に留めてよい
3. 文献数が多い(目安 30 件超)場合は、優先度順の部分検証に切り替える旨を
   メイン経由でユーザーに報告する

## 2. 引用の使い方(帰属の正しさ)

- **原論文が実際に主張したことと本文の帰属が一致するか**: abstract を読んで
  照合する。実例: 「HFT の急変時退出を実証した」と書かれていたが、原論文の
  abstract は「取引パターンは変化しなかった」— 正反対の帰属だった
- 「論文が実証したこと」と「論文から着想を得た実装」の区別。理論との直結表現
  (「X の理論に基づく」)は「X の考え方を参考にした簡略化」等へ格下げが必要な
  ことが多い
- 「◯◯は 3 つある」等の限定表現: 原典がその数に限定しているか(原典の整理が
  より広いなら「このうち本稿は上記 3 つを対象とする」に直す)
- 引用なしの一般化(「近年議論されている」「広く知られている」)への引用付与

## 3. 本文の数値とデータ・実装の突合

原稿中の全数値(検定統計量、p 値、n、比率、閾値、seed 数)をリポジトリ内の
根拠と照合する:

- 結果ファイル(CSV / JSON)の値と一致するか。丸め方は一貫しているか
- 設定ファイル(config 等)と実験条件の記述(seed 数、パラメータ、条件数)が
  一致するか
- 数値の転記ミスだけでなく「その数値を生んだ手続き」まで遡る。手続きの妥当性
  (検定の選択が実験設計に合うか等)は methods 観点の担当だが、突合中に設定
  ファイルとの不整合に気づいたら報告してよい
- 図の元データと本文の記述(判定範囲・集約方法)が一致するか

## 4. 指摘の書き方

- 根拠欄には自分が実際に開いた URL / file:line を書く。「確認した」とだけ
  書かない
- 原典に当たれなかった(paywall 等)場合は指摘を出してよいが、根拠欄に
  「未確認 — abstract のみ / アクセス不可」と明記して重大度を下げる
